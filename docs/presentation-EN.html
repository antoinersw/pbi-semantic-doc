04  —  The TMDL Parser
What is extracted
from the semantic model
TMDL Object What we extract
Table → Name, description, hidden, lineage tag
Column → Type, format, description, hidden, calculated
Measure → DAX expression, display_folder, format string
Relationship → From/To table+column, cardinality, cross_filter, is_active
Partition (M / SQL) → Source expression, query folding, steps Power Query
RLS Role → Role name, filter expressions per table
The PBIR format uses separate .tmdl files for each object.
The parser recursively reads the definition/ folder of the SemanticModel.
Folder structure
📁 MyModel.SemanticModel/

    📁 tables/
      📄 Sales.tmdl
      📄 DimCalendar.tmdl
    📁 relationships/
    📁 roles/
  📁 .pbi/

M Code parser

Extracts Power Query steps, detects query folding and reconstructs native SQL query if present.

06  —  Measure Lineage
The directed reverse-filter graph — who can filter whom?
✅ Compatible
DimCalendar · DimCliente · DimProdotto
BFS ancestors of FAT_Sales
❌ Non correlato
FAT_Acquisti
another FAT table
⚠️ Filtro rimosso
ALL(DimCalendar)
removed in measure via ALL()
The reasoning
01
Power BI Relationships
FAT→DIM (many-to-one): filters flow from DIM to FAT. FAT cannot filter other FAT tables.
02
Reverse-filter graph
Archi FAT→DIM (inverse of filter flow). BFS from FAT reaches only the correct ancestors.
03
BFS = compatible slicers
Nodes reachable via BFS = esattamente exactly the tables that can filter the measure.
04
Bidirectional
Se Both, edge in both directions. Useful but handle with care.
"Questo è il pezzo più interessante. Molti tool mostrano le relazioni, ma nessuno risponde alla domanda: 'Se uso questa misura in un visual, quali dimensioni posso mettere come slicer senza che i dati siano sbagliati?' La risposta è la BFS sul grafo reverse-filter."

06b  —  BFS — Breadth-First Search
Explore the graph level by level
All direct neighbours first, then go deeper
Start
visited = {"FAT_Sales"}    queue = ["FAT_Sales"]
FAT_Sales
Step 1
pop "FAT_Sales" → neighbours: DimCalendar, DimCliente, DimProdotto
DimCalendarDimClienteDimProdotto
Step 2
pop "DimCalendar" → no unvisited neighbours
✓ DimCalendar
Step 3
pop "DimCliente" → neighbour: BridgeTable (level 2)
BridgeTable
✅
Compatible slicers for [Total Sales All Time]
visited = { FAT_Sales, DimCalendar, DimCliente, DimProdotto, BridgeTable }
BFS vs DFS
BFSQueue (FIFO). Visits all nodes at level N before going to N+1. Guarantees shortest path.
DFSStack (LIFO). Goes deep first. Same result for our use case, but less intuitive.
Complexity
O(V+E)
V = nodes (tables)
E = edges (relationships)
Real Power BI models: V ≈ 10–50, E ≈ 10–80.
Practically instantaneous.

"BFS is a classic algorithm — you'll find it in any algorithms textbook. We use it because it guarantees visiting all reachable nodes in the reverse-filter graph, regardless of depth. On a typical Power BI model with 10–50 tables it's practically instantaneous."

09  —  Installation & Usage
Three commands.
Documentation ready.
# 1. Installation
pip install pbi-semantic-doc

# 2. Semantic Model only → HTML + MD
pbi-semantic-doc "MyModel.SemanticModel"

# 3. Full .pbip project (Semantic Model + Report)
pbi-semantic-doc "MyProject.pbip" --combined
# or: parent folder containing both
pbi-semantic-doc "./MyProject" --combined

# Generated output:
DOC_MyModel.html ← open in browser
DOC_MyModel.md ← commit to git
# Or via Python API:
from pbi_semantic_doc.parser import TmdlParser
from pbi_semantic_doc.html_generator import HtmlGenerator

model = TmdlParser().parse( Path("MyModel.SemanticModel/definition") )
html = HtmlGenerator().generate( model )
Path("DOC_MyModel.html").write_text( html )
