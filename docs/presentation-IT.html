04  —  Il Parser TMDL
Cosa viene estratto
dal modello semantico
Oggetto TMDL Cosa estraiamo
Table → Nome, descrizione, hidden, lineage tag
Column → Tipo, format, descrizione, hidden, calcolata
Measure → DAX expression, display_folder, format string
Relationship → From/To table+column, cardinality, cross_filter, is_active
Partition (M / SQL) → Source expression, query folding, steps Power Query
RLS Role → Nome ruolo, espressioni filtro per tabella
La struttura PBIR usa file .tmdl separati per ogni oggetto.
Il parser legge ricorsivamente la cartella definition/ del SemanticModel.

05  —  Misure & DAX Analyzer
Cosa sa il sistema
di ogni misura DAX
# extract_dax_refs() ritorna:
@dataclass
class DaxRefs:
  aggregated_tables: set[str] # COUNTROWS, SUM, etc.
  measure_refs: set[str] # [OtherMeasure] refs
  table_column_refs: list[tuple] # Table[Column] refs
  uses_time_intel: bool # DATEADD, SAMEPERIOD…
  removes_filters: bool # ALL, REMOVEFILTERS…
  uses_calculate: bool # CALCULATE presente

# Esempio concreto:
Total Sales = CALCULATE(
  SUM( Sales[Amount] ),
  ALL( DimCalendar )
)

→ aggregated_tables: {'Sales'}
→ table_column_refs: [('Sales','Amount'), ('DimCalendar',*)]
→ removes_filters: True
→ uses_calculate: True

06  —  Lineage delle Misure
Il grafo diretto reverse-filter — chi può filtrare chi?
✅ Compatible
DimCalendar · DimCliente · DimProdotto
ancestor BFS di FAT_Sales
❌ Non correlato
FAT_Acquisti
altra tabella FAT
⚠️ Filtro rimosso
ALL(DimCalendar)
la misura ignora il filtro data
🔍 Come funziona BFS + grafo reverse-filter — clicca per espandere
Struttura dati
graph: dict[str, set[str]]
# es:
{ "FAT_Sales": {"DimCalendar",
                   "DimCliente",
                   "DimProdotto"} }
BFS implementation
visited = {"FAT_Sales"}
queue = ["FAT_Sales"]
while queue:
  node = queue.pop(0)
  for n in graph[node]:
    visited.add(n) # DIM!
Il ragionamento
01
Relazioni Power BI
FAT→DIM (many-to-one): il filtro scorre da DIM verso FAT. FAT non filtra altre FAT.
02
Grafo reverse-filter
Archi FAT→DIM (inversi al filtro). BFS da FAT risale ai soli antenati corretti.
03
BFS = compatible slicers
Nodi raggiungibili con BFS = esattamente le tabelle che possono filtrare la misura.
04
Bidirezionale
Se Both, arco in entrambe le direzioni. Utile ma da usare con cautela.
"Questo è il pezzo più interessante. Molti tool mostrano le relazioni, ma nessuno risponde alla domanda: 'Se uso questa misura in un visual, quali dimensioni posso mettere come slicer senza che i dati siano sbagliati?' La risposta è la BFS sul grafo reverse-filter."

06b  —  BFS — Breadth-First Search
Esplora il grafo livello per livello
Tutti i vicini diretti prima di scendere in profondità
Start
visited = {"FAT_Sales"}    queue = ["FAT_Sales"]
FAT_Sales
Step 1
pop "FAT_Sales" → vicini: DimCalendar, DimCliente, DimProdotto
DimCalendar DimCliente DimProdotto
Step 2
pop "DimCalendar" → nessun vicino non visitato
✓ DimCalendar
Step 3
pop "DimCliente" → vicino: BridgeTable (livello 2)
BridgeTable
✅
Compatible slicers per [Total Sales All Time]
visited = { FAT_Sales, DimCalendar, DimCliente, DimProdotto, BridgeTable }
BFS vs DFS
BFS Coda (FIFO). Visita tutti i nodi al livello N prima di scendere al livello N+1. Garantisce il cammino minimo.
DFS Stack (LIFO). Scende in profondità prima di risalire. Risultato identico per il nostro caso, ma meno intuitivo.
Complessità
O(V+E)
V = nodi (tabelle)
E = archi (relazioni)
Su modelli Power BI reali: V ≈ 10–50, E ≈ 10–80.
Praticamente istantaneo.
📚 Riferimenti
Introduction to Algorithms
Cormen, Leiserson, Rivest, Stein (CLRS)
Cap. 22.2 — Breadth-first search
Wikipedia — BFS
en.wikipedia.org/wiki/Breadth-first_search
Python collections.deque
docs.python.org — O(1) popleft per BFS efficiente
"BFS è un algoritmo classico — lo trovate in qualsiasi libro di algoritmi. Noi lo usiamo perché garantisce di visitare tutti i nodi raggiungibili nel grafo reverse-filter, indipendentemente dalla profondità. Su un modello Power BI tipico con 10-50 tabelle è praticamente istantaneo."

09  —  Installazione & Uso
Tre comandi.
Documentazione pronta.
# 1. Installazione
pip install pbi-semantic-doc

# 2. Solo Semantic Model → HTML + MD
pbi-semantic-doc "MyModel.SemanticModel"

# 3. Progetto .pbip (Semantic Model + Report)
pbi-semantic-doc "MyProject.pbip" --combined
# oppure: cartella parent che contiene entrambi
pbi-semantic-doc "./MyProject" --combined

# Output generato:
DOC_MyModel.html ← apri nel browser
DOC_MyModel.md ← commit su git
# Oppure via Python API:
from pbi_semantic_doc.parser import TmdlParser
from pbi_semantic_doc.html_generator import HtmlGenerator

model = TmdlParser().parse( Path("MyModel.SemanticModel/definition") )
html = HtmlGenerator().generate( model )
Path("DOC_MyModel.html").write_text( html )
